01What changed in your code

The deprecated oldFetch(url, method, body, headers) was replaced by newFetch(url, opts) at every callsite — 22 of them, across 13 files.

This was never a find-and-replace. The new signature takes a typed RequestOpts options object instead of four positional arguments, and — the wrinkle that makes each edit a small piece of reasoning — newFetch returns Promise<Response> and no longer auto-parses JSON. So every callsite needed three coordinated changes: rewrite the call into the options shape, fix the import, and adjust whatever the code did with the awaited result — usually adding a .json() (or .blob() / .text()) unwrap that oldFetch used to do for you. The table below is every one of those edits, with file:line and before→after, so you can open each and check the positional-to-named mapping by eye.

The one callsite that couldn’t be ported mechanically — flushEvents, which passes a 5th positional argument the old signature never had — shows as the BLOCKED row with no success diff, and is broken out as a decision for you further down.

The call is oldFetch(endpoint, opts.method, opts.payload, opts.headers, opts.beacon) — five positional arguments against a documented four-arg signature (url, method, body, headers). The trailing opts.beacon (a boolean) maps to nothing. It looks like a request for sendBeacon-style fire-and-forget delivery, which newFetch models differently, or maybe not at all. The decision needs a human: how should opts.beacon === true be expressed under the new API?

keepalive flag · recommended, pending confirmation

Replace the call with navigator.sendBeacon(endpoint, opts.payload) when beacon is true, else fall back to newFetch. Matches original fire-and-forget semantics but changes control flow and drops headers/method.

Topology, why-this-shape, and the run accounting

The shape, in three sentences. A single ripgrep counted the work first — 23 hits across 14 files — and dispatch fanned out one worker per file (not per callsite), because independence is at the file/merge boundary: two edits in the same file share a buffer and imports, so each file gets its own git worktree to keep same-file edits serialized while different files run in parallel. The workers are cheap and untrusted; the verifier is the deterministic spine — build exit code plus test exit code, no model judging anything — so the whole batch converges at one barrier that merges the clean branches and hands a coherent integration branch to that verifier, the only node allowed to say “done.” The blocked branch peels off before the barrier: w13 committed to its ambiguous line, was excluded from the merge, and so let the verifier run green on a coherent subset of 13 files instead of failing the whole batch on one unanswerable question.
