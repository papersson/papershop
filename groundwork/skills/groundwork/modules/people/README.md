# people — whose metric the tool serves versus whose hands it lives in

Mid-level index (progressive disclosure). Enter when the commissioner who asked for the tool is not the operator who will run it, and you need to know whether it lands as enablement or as surveillance before rollout.

## Leaves
- `role-interest-map.md` — segment by organizational function keyed to the metric each role is measured on; surface the operator-versus-commissioner divergence and the route-around risk before anyone builds. [CORE]

## Fit
Second stop in the SKILL.md interview-assist loop on the default order, but pulled forward the moment the brief shows a commissioner buying a tool for someone else's hands — the configuration where adoption quietly dies at rollout. This branch is deliberately one leaf. Veto and block power is not a separate artifact; it is a **column on the role map**, sitting next to the metric that explains it, and the divergence and route-around columns that make it actionable. Splitting blocker power into its own RACI-style leaf would buy a second table that duplicates the map's keys and reads worse than the one row where a role's veto, its metric, and its likely shadow system already line up — exactly the artifact-proliferation the skill is built to avoid. The branch takes the named human dependencies the as-is workaround register surfaced, attaches a role and a metric to each, and hands the frame branch the tension it must resolve: when an operator's friction contradicts a commissioner's stated goal, whose interest does the first slice serve.
